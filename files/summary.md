# Cipla Execution Intelligence Platform - Simple Summary

## Main Goal

Build a serious internal analytics app for Cipla's EMEU/PBP execution data.

The app will take messy Excel/XLSB workbooks and turn them into a trusted dashboard that answers:

- which planned activities were executed, delayed, missed, or unmatched,
- where budget was planned, confirmed, spent, underused, or overspent,
- where requests are stuck in approval, confirmation, or post-event reporting,
- which intervention types consume budget and effort,
- which doctors were engaged and whether they prescribe Cipla brands,
- which high-prescribing doctors are not being engaged,
- what data-quality issues make a metric risky to trust,
- what business summary can be safely generated by AI from trusted data.

This is not a spreadsheet viewer. It is a data product with ingestion, validation, reconciliation, database views, APIs, dashboard pages, and grounded AI summaries.

## What The App Will Look Like

The final app will have:

- an Executive Overview page,
- an Execution Matrix page,
- a Budget Utilization page,
- a Doctor ROI page,
- a Data Quality page,
- a grounded AI assistant panel,
- global filters for country, month, therapy, event type, intervention type, doctor class, and ROI segment.

Every page must show loading, empty, error, stale-data, weak-match, missing-FX, no-RCPA, and partial-data states where relevant.

## Source Data Used

The app uses these workbook families:

- RCPA files for doctor, brand, SKU, prescription quantity/value, own-vs-competitor behavior, Pcodes, and doctor status.
- Yearly planner files for planned events, planned HCPs, therapies, brands, event types, and planned budget.
- Consolidation report for actual execution requests, confirmed/contracted amount, actual spend, BTU/BTC spend split, attendance, approval status, reporting status, and intervention type.
- Monthly execution planner files for monthly execution snapshots, raised requests, approved doctors, executed/action-due status, and YP totals.

## Core Architecture

The data flow is:

```text
Excel / XLSB files
  -> local Python ingestion CLI
  -> workbook profiling
  -> validation and normalization
  -> Supabase PostgreSQL canonical tables
  -> explicit reconciliation tables
  -> materialized KPI views
  -> FastAPI backend
  -> React dashboard
  -> grounded AI assistant
```

## Important Design Decisions

- Raw Excel files stay local and are not committed to git.
- Ingestion is local CLI-based for MVP, not browser upload.
- Supabase PostgreSQL is used as the database.
- The frontend does not talk directly to Supabase.
- FastAPI is the only dashboard API layer.
- KPI math is done in SQL/materialized views or typed backend services, not in frontend code and not by AI.
- AI only summarizes trusted backend results and must refuse unsupported questions.
- RCPA rows are aggregated before database persistence.
- Pcodes are stored as text, not numbers.
- Currency values preserve local currency; Sri Lanka/LKR USD values use the official company rate `1 USD = 310 LKR`, and other USD values are used only when FX rates exist.
- Temporary FX rates for non-LKR currencies are allowed but must be marked `provisional`; LKR is official at `1 USD = 310 LKR`.

## Supabase Answer

Yes, the project uses Supabase.

More precisely, it uses Supabase PostgreSQL as the hosted database. The app should not use Supabase as a direct frontend data client. The ingestion CLI and FastAPI backend access the database server-side, while the React frontend only calls FastAPI.
